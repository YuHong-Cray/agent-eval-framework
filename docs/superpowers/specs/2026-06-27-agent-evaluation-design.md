# Coding Agent 深度评测体系设计

> 版本: 1.0 | 日期: 2026-06-27 | 状态: 已批准

---

## 1. 概述

### 1.1 目标

建立一套标准化的 Coding Agent 评测体系，用于：
- 量化评估 Coding Agent 的核心能力
- 横向对比不同 Agent 产品（Claude Code、Copilot、Cursor 等）
- 持续追踪 Agent 版本迭代中的能力变化
- 定位 Agent 在具体维度上的优劣势

### 1.2 设计原则

- **可量化**：每个维度有明确的数字指标
- **可复现**：相同条件下重复测试，结果一致
- **可对比**：统一的测题集和执行环境，公平横向对比
- **可扩展**：测题集和维度可按需追加

---

## 2. 评测维度

### 2.1 六大核心维度

| 编号 | 维度 | 核心定义 | 关键子能力 |
|------|------|---------|-----------|
| D1 | 代码生成 | 根据需求描述生成正确、可维护、符合规范的代码 | 语法正确性、逻辑完备性、边界处理、代码风格/可读性、测试覆盖 |
| D2 | 任务拆解 | 将复杂需求分解为合理的子任务和执行顺序 | 拆解粒度合理性、依赖识别、优先级排序、与可用工具的匹配 |
| D3 | 工具调用 | 在合适时机选择合适工具，正确传参，处理返回值 | 工具选择准确性、参数正确性、错误处理、链式调用编排 |
| D4 | 多智能体协作 | 将子任务分派给子 agent，合并结果，处理冲突 | 分派合理性、结果整合质量、一致性校验、协作效率 |
| D5 | 审查/调试 | 发现并修复代码缺陷，定位根因，给出改进建议 | Bug 发现率、根因定位准确度、修复正确性、建议质量 |
| D6 | 持续记忆 | 跨会话/跨轮次保持上下文，从反馈中学习调整 | 上下文保持度、反馈响应正确性、偏好积累、错误不再犯 |

## 3. 三层评测架构

### 3.1 架构总览

```
Layer 3 ██ 长程项目测试
         题目数: 2-4 | 时长: 每项目跨多会话
         判分: LLM-as-Judge + 人工抽检
         重点: D6(记忆) D4(多智能体) D2(拆解)

Layer 2 ██ 集成场景测试
         题目数: 8-12 | 时长: 每题30-60min
         判分: LLM-as-Judge（结构化评分标准）
         重点: 全部6维融合作战

Layer 1 ██ 原子能力基准
         题目数: 40-60题 | 时长: 每题5-15min
         判分: 自动化（单测通过+工具轨迹匹配）
         重点: D1 D3 D5 可量化项
```

### 3.2 各维度在分层中的覆盖权重

| 维度 | Layer 1 | Layer 2 | Layer 3 |
|------|---------|---------|---------|
| D1 代码生成 | ★★★ 核心 | ★★ 辅助 | ★ 低度 |
| D2 任务拆解 | ★★ 辅助 | ★★★ 核心 | ★★ 辅助 |
| D3 工具调用 | ★★★ 核心 | ★★ 辅助 | ★ 低度 |
| D4 多智能体 | ★ 低度 | ★★★ 核心 | ★★★ 核心 |
| D5 审查调试 | ★★★ 核心 | ★★ 辅助 | ★ 低度 |
| D6 持续记忆 | ★ 低度 | ★ 低度 | ★★★ 核心 |

## 4. 测题设计

### 4.1 Layer 1：原子能力基准（40-60题）

每题是独立的、可自动化判分的原子任务。

#### 题型一览

| 题型 | 覆盖维度 | 示例 | 判分方式 |
|------|---------|------|---------|
| 代码填空 | D1 | 给定函数签名和 docstring，补全实现 | 单元测试通过率 + 覆盖率 |
| Bug 定位 | D5 | 给定含 1-3 个 bug 的代码文件，要求找出并修复 | bug 检出率 + 修复正确率 |
| 工具选择 | D3 | 描述一个操作需求，让 agent 选择正确工具及参数 | 工具名+参数精确匹配 |
| 子任务拆解 | D2 | 给定复杂需求，要求输出子任务列表及依赖图 | 与标准拆解树的结构相似度 |
| 代码审查 | D5 | 给定 PR diff，要求给出审查意见 | 关键问题点命中率 |
| 链式工具编排 | D3 | 需要连续调用 2-4 个工具才能完成的任务 | 工具序列正确性 |

#### 判分自动化脚本

- L1-D1/D5：运行单元测试（pytest / jest / go test），统计通过率
- L1-D3：正则/JSON 结构匹配工具调用轨迹
- L1-D2：计算 Agent 输出的任务树与标准树之间的树编辑距离

### 4.2 Layer 2：集成场景测试（8-12题）

每道题是一个完整的开发微场景，agent 需要完成从需求理解到交付的全流程。

#### 场景设计原则

- 每个场景天然需要 3+ 个维度的能力
- 有明确的"交付物"（可运行代码 + 测试 + 文档）
- 必须经过任务拆解才能有效完成
- 内含"陷阱"和"歧义"——考验 Agent 主动澄清或合理假设的能力

#### 场景模板示例

```
场景: "给这个博客系统添加搜索功能"
  输入: 现有项目代码 + 模糊需求描述
  期望交付: 搜索 API + 前端搜索框 + 测试 + 使用说明
  隐含难点: 理解现有数据模型、选择搜索方案（LIKE vs 全文索引）、处理中文分词
  评分维度: D1(代码质量) + D2(拆解) + D3(文件读写、命令执行) + D4(可选子agent分派)
```

#### 评分方式

使用统一 LLM-as-Judge 评分卡，每个场景在六个维度上各打 1-5 分，附带评分理由。

### 4.3 Layer 3：长程项目测试（2-4题）

跨多个会话的持续性项目，验证记忆保持和多智能体协作深度。

#### 题目设计

| 项目 | 导向 | 设计 |
|------|------|------|
| 项目 1 | 记忆导向 | 连续 3 个会话完成一个功能迭代，验证上下文保持 |
| 项目 2 | 协作导向 | 大型重构，必须使用子 agent 分工完成 |
| 项目 3 | 混合导向 | 从零开发小型应用，跨多天完成 |

#### 评分方式

- LLM-as-Judge 对每个会话产出评分
- 跨会话能力衰减曲线分析
- 人工抽检 20% 样本校准

## 5. 评分体系

### 5.1 总评分公式

```
最终得分 = L1 × 0.40 + L2 × 0.40 + L3 × 0.20

其中：
  L1 = Σ(各题型得分 × 题型权重) / 总题数
  L2 = Σ(各场景六维评分) / 场景数
  L3 = Σ(各项目评分) / 项目数
```

### 5.2 判分方式

| 层级 | 判分方式 | 说明 |
|------|---------|------|
| Layer 1 | 自动化判分 | 单元测试通过率、工具匹配率、拆解树相似度 |
| Layer 2 | LLM-as-Judge | DeepSeek-V4-Flash 独立实例，结构化评分卡 |
| Layer 3 | LLM-as-Judge + 人工 | DeepSeek-V4-Flash + 20% 样本人工抽检校准 |

### 5.3 LLM-as-Judge 质量保障

- 使用独立的 DeepSeek-V4-Flash 实例，与评测对象隔离，避免自评偏差
- 评分卡采用 JSON Schema 约束输出，确保结构化
- 人工抽检 20% 样本，计算 judge 偏差和一致性系数
- 偏差超过阈值（>10%）时触发 judge 重新校准

## 6. 测试执行基础设施

### 6.1 系统架构

```
┌──────────────────────────────────────────────────┐
│              评测编排层 (Orchestrator)              │
│  负责调度题目、注入上下文、收集输出、触发判分        │
├──────────────────────────────────────────────────┤
│  被测 Agent 接口层 (Agent Adapter)                  │
│  统一接口适配不同 agent (Claude Code / Copilot / ..) │
├──────────────┬───────────────────────────────────┤
│  沙箱执行环境  │  判分服务 (Scoring Engine)          │
│  Docker 隔离  │  L1 自动化判分 + L2/L3 LLM-Judge   │
├──────────────┴───────────────────────────────────┤
│  结果存储 & 可视化 (Results DB + Dashboard)         │
└──────────────────────────────────────────────────┘
```

### 6.2 组件详解

#### 编排层 (Orchestrator)

- 按顺序或并行调度测题执行
- 为每道题准备独立的 Docker 沙箱环境
- 注入题目前置条件（代码仓库、依赖、环境变量）
- 收集 Agent 的完整行为轨迹（每步工具调用、输出、耗时）
- 将轨迹和产出物送交判分服务
- 处理超时、重试、异常中断

#### Agent 接口层 (Agent Adapter)

统一接口：

```typescript
interface AgentAdapter {
  // 发送 prompt 并获取完整执行轨迹
  execute(prompt: string, context: TestContext): Promise<AgentTrace>;

  // 获取 agent 能力描述
  capabilities(): AgentCapabilities;
}
```

需要实现的适配器：
- `ClaudeCodeAdapter` — 通过 CLI / SDK 驱动
- `CopilotAdapter` — 通过 VS Code 扩展 API 驱动
- `CursorAdapter` — 类似方案
- 通用 `CLIAdapter` — 适配任何命令行 agent

#### 沙箱执行环境

- Docker 容器，预装对应语言运行时和依赖
- 每次测题复用或快速重建干净环境
- 网络隔离选项：禁止 / 白名单 / 全开放 三档
- 文件系统快照：执行前后对比，检测文件操作正确性

#### 判分服务 (Scoring Engine)

- **Layer 1 判分**：单元测试运行器、工具调用轨迹匹配器、拆解树相似度计算
- **Layer 2/3 判分**：LLM-as-Judge（DeepSeek-V4-Flash 独立实例）、结构化评分卡输出、评分校准

#### 结果存储与可视化 (Results DB + Dashboard)

- 每次运行生成唯一 run_id，关联 agent + 题目 + 时间
- 存储完整行为轨迹（可回放分析）
- Dashboard：六维雷达图、横向对比柱状图、单题得分下钻明细

### 6.3 技术选型

| 组件 | 推荐方案 | 备选 |
|------|---------|------|
| 编排 | Python (pytest 框架扩展) | Node.js |
| 沙箱 | Docker + docker-py | Podman |
| 判分 DB | PostgreSQL (JSONB 存轨迹) | SQLite |
| 可视化 | Streamlit / Grafana | 自建前端 |
| LLM-Judge | DeepSeek-V4-Flash (独立实例) | — |

## 7. 评测流程与运行策略

### 7.1 单次完整评测流程

```
准备阶段
  ├─ 确认被测 agent 版本/配置
  ├─ 拉取最新测题集
  ├─ 构建/预热 Docker 镜像
  └─ 初始化判分服务

Layer 1 执行 (≈2-3小时)
  ├─ 并行运行 40-60 道原子题（受 Docker 沙箱并发上限约束）
  ├─ 每题: 启动沙箱 → 注入题目 → agent 执行 → 收集轨迹 → 自动化判分
  └─ 生成 L1 维度得分矩阵

Layer 2 执行 (≈8-12小时，可分批)
  ├─ 串行或低并发运行 8-12 个集成场景
  ├─ 每题: 启动沙箱 → 注入项目+需求 → agent 完成交付 → 收集产物+轨迹
  ├─ LLM-Judge 对每题产出打分（异步批处理）
  └─ 生成 L2 六维评分矩阵

Layer 3 执行 (跨数天)
  ├─ 按顺序执行 2-4 个长程项目
  ├─ 每个项目跨 3-5 个会话，记忆评估项需要在会话间追踪
  └─ 人工 + LLM-Judge 联合评分

总评计算
  ├─ 加权汇总三层得分
  ├─ 生成评测报告（含雷达图 + 横向对比 + 下钻明细）
  └─ 归档到结果数据库
```

### 7.2 运行策略

| 维度 | Layer 1 | Layer 2 | Layer 3 |
|------|---------|---------|---------|
| 并发策略 | 高并发（按 CPU 核数） | 低并发（2-3题） | 串行 |
| 重试策略 | 每题重试 1 次（超时/异常） | 不重试 | 不重试 |
| 超时 | 15 分钟/题 | 60 分钟/题 | 120 分钟/会话 |
| 执行频率 | 每次版本发布必跑 | 每次版本发布必跑 | 重大版本或月度 |

## 8. 测试题集管理与扩展

### 8.1 题目元数据标准

每道题携带统一元数据：

```json
{
  "id": "L1-D1-PY-001",
  "layer": "L1",
  "dimensions": ["D1"],
  "sub_dimensions": ["D3"],
  "language": "python",
  "difficulty": 2,
  "estimated_time_min": 8,
  "tags": ["function", "string-manipulation"],
  "created": "2026-07-01",
  "version": "1.0",
  "sandbox": {
    "image": "python:3.12",
    "dependencies": ["pytest"],
    "network": "none"
  },
  "prompt_template": "...",
  "context_files": ["src/main.py", "tests/test_main.py"],
  "expected_artifacts": ["src/main.py"],
  "scoring": {
    "type": "unit_test",
    "test_command": "pytest tests/",
    "pass_threshold": 0.85
  }
}
```

### 8.2 题目来源

| 来源 | 占比 | 说明 |
|------|------|------|
| 人工设计核心库 | 60% | 团队精心设计，覆盖各维度的典型和边界场景 |
| 开源项目改编 | 25% | 从 GitHub 真实 issue/PR 改编，保证真实性 |
| 对抗生成 | 15% | 使用强 agent 自动生成候选题目，人工筛选校验 |

### 8.3 题目迭代与防污染

- **版本化管理**：每道题带版本号，修改自动升版本，旧版本保留
- **轮换池机制**：总题库 > 单次测试量，每次随机抽取
- **对抗更新**：定期分析高频错误模式，针对性补充边界测题
- **保密性**：Layer 2 和 Layer 3 核心题目不公开，仅评测团队可见

## 9. 横向对比框架

### 9.1 变量控制

| 变量 | 控制方式 |
|------|---------|
| 测题一致 | 同一版本测题集，同一随机种子抽题 |
| 环境一致 | 同一 Docker 镜像，同一网络条件 |
| 时间窗口 | 所有 Agent 在 72 小时内完成同一轮评测 |
| 模型版本 | 记录每个 Agent 使用的底层模型配置 |
| 提示词 | 统一入口 prompt 格式，不额外注入偏好 |

### 9.2 对比报告模板

```
┌─────────────────────────────────────────────────────┐
│               Agent 横向对比报告 vYYYY.MM             │
├─────────────────────────────────────────────────────┤
│  总分排名:                                            │
│  1. Agent A  87.3  (代码 91, 拆解 85, 工具 89, ...)   │
│  2. Agent B  82.1  (代码 79, 拆解 88, 工具 81, ...)   │
│  3. Agent C  76.5  (...)                             │
│                                                       │
│  维度雷达图: [六维叠加对比]                             │
│  优劣势分析:                                          │
│  能力曲线: [Layer 1/2/3 各层级得分趋势]                │
│  改进建议:                                            │
└─────────────────────────────────────────────────────┘
```

### 9.3 对比频率

| 场景 | 频率 | 范围 |
|------|------|------|
| 例行对比 | 月度 | Layer 1 + Layer 2 全量 |
| 版本对比 | 发版时 | 新版本 vs 上一版本 vs 竞品最新版 |
| 深度对比 | 季度 | 三层全量 + 人工深度评审 |
| 专项对比 | 按需 | 聚焦某个维度的深入对比 |

## 10. 实施路线建议

### 阶段一：基础设施搭建（2-3周）
- 搭建 Docker 沙箱和编排框架
- 实现 1-2 个 Agent Adapter（优先 ClaudeCodeAdapter 和 CLIAdapter）
- 设计 Layer 1 首批 20 题 + 判分脚本

### 阶段二：核心题库建设（4-6周）
- 完成 Layer 1 全部 40-60 题
- 设计 Layer 2 首批 4 场景
- 搭建 LLM-as-Judge 评分流程

### 阶段三：完整体系上线（2-3周）
- 完成 Layer 2 全部 8-12 场景
- 设计 Layer 3 首批 2 项目
- 部署 Dashboard 可视化

### 阶段四：持续运营
- 补充题目、对抗更新
- 接入更多 Agent 产品
- 定期产出对比报告

## 附录A：判分标准参考

### L1 题型权重

| 题型 | 权重 | 满分条件 |
|------|------|---------|
| 代码填空 | 0.25 | 单元测试通过率 ≥ 85% |
| Bug 定位 | 0.20 | Bug 检出率 100% + 修复正确率 ≥ 80% |
| 工具选择 | 0.15 | 工具名正确 + 关键参数正确 |
| 子任务拆解 | 0.15 | 拆解树与标准树编辑距离 ≤ 阈值 |
| 代码审查 | 0.15 | 关键问题点命中率 ≥ 70% |
| 链式工具编排 | 0.10 | 工具序列完全正确 |

### L2/L3 评分卡维度（1-5分制）

每个维度评分标准：
1. 未达预期 — 严重缺陷，交付物不可用
2. 勉强可用 — 核心功能部分实现，但存在明显问题
3. 达到基准 — 核心功能正确实现，无严重缺陷
4. 超出预期 — 实现全面，边界处理得当，代码质量好
5. 卓越 — 产出可视为最佳实践范例，展现深度理解
